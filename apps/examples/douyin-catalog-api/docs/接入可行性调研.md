> 读完此文档,业务/技术能联合决策走下一步:Step 1 开干、走变体方案、或 pivot 到其他平台。

## 一句话

> **技术上做得了,但准入门槛是真正阻塞**。抖音精选联盟 API 历来是机构 / MCN / 团长资质准入,
> 个人或普通公司账号大概率拿不到 API 权限,只能用网页手动选品(无法做 OCP 自动化)。
> **第一优先级是业务侧解决准入,而不是技术侧动手写代码**。

## 一、抖音电商生态 — 4 个子平台先分清

很多人把"抖音电商"当成一个东西,实际上是 **4 个子平台 + 1 个开发者门户**:

| 平台 | URL | 服务对象 | 与 OCP 关系 |
| --- | --- | --- | --- |
| **抖音开放平台** | developer.open-douyin.com | 所有开发者(App接入/小程序等) | 鉴权基础(OAuth 入口) |
| **精选联盟 / 巨量百应** | buyin.jinritemai.com | 达人 / MCN / 团长(流量方) | ⭐ **OCP 要接的就是这个** |
| **抖音小店 / ISV** | fxg.jinritemai.com | 商家 / SaaS 服务商 | 商家维度(自营店),与 OCP catalog node 角色不符 |
| **巨量千川** | qianchuan.jinritemai.com | 商家付费投放 | CPM/CPC 广告,**不是 CPS**,与 OCP 模型不符 |
| **抖音橱窗** | App 内 | 个人达人 | 个人选品工具,**没有开放 API** |

**结论**:OCP 接入抖音 = 接入 **精选联盟**。其他 3 个子平台的 API 都不解决我们的问题。

## 二、精选联盟 API 准入条件分级

⚠️ 以下信息基于公开渠道整理,**真实准入条件随抖音政策调整可能变化**,正式申请前需要在开放平台核实最新规则。

| 主体类型 | 准入难度 | 能拿到的 API | 是否够 OCP 用 |
| --- | --- | --- | --- |
| **个人达人账号**(粉丝<1000) | ✅ 一注册就有 | ❌ 只能用网页 UI 手动选品,**无 API** | ❌ 完全不够 |
| **个人达人**(粉丝>1000 + 完成实名) | ✅ 容易 | 🟡 部分基础 API(如商品浏览),**精选联盟核心 API 多需要额外申请** | ⚠️ 大概率不够 |
| **公司主体 + 普通账号** | 🟡 中等 | 🟡 类似个人,**核心 API 没有专属通道** | ⚠️ 大概率不够 |
| **MCN 机构号**(认证) | 🔴 高(需要旗下达人 + 营收门槛) | ✅ 完整精选联盟 API | ✅ 够 |
| **团长账号**(认证) | 🔴 中-高(需要现有联盟流量基础) | ✅ 团长侧完整 API | ✅ 够 |
| **官方 ISV 服务商** | 🔴 高(需要营业执照 + 服务能力评估) | ✅ 给商家用的 API,**不是给推荐方的** | ❌ 角色不符 |
| **字节 BD 绿色通道** | 🔴 极高(需要业务规模背书) | ✅ 任意定制权限 | ✅ 够 |

**关键判断**:OCP 想要的 4 类 API(物料搜索 / 转链 / 订单查询 / 增量订单)**至少需要 MCN 或团长级别准入**。

## 三、鉴权:OAuth 2.0 (与前 3 家根本性不同)

这是最大的架构差异点,直接决定 Step 2 client 代码不能套 alimama/JD/PDD 模板。

### 抖音 vs 前 3 家鉴权对比

| 维度 | alimama / JD / PDD | **抖音** |
| --- | --- | --- |
| 鉴权方式 | MD5 一次性签名 | **OAuth 2.0 access_token** |
| 凭据 | appKey + appSecret(永久) | **client_key + client_secret + access_token + refresh_token** |
| 调用前要做什么 | 把 sysParams 排序签名 | **检查 token 是否过期、过期了用 refresh_token 刷新** |
| Token 有效期 | 永久 | access_token 通常 **2 小时**,refresh_token **30 天** |
| 维度 | 应用维度(配置一次永久用) | **用户维度**(每个 MCN / 团长账号要单独 OAuth 授权一次) |
| 失败模式 | 签名错误 = 改下次 | **Token 失效要无感刷新,否则整个服务挂掉** |

### OAuth 2.0 标准流程(抖音版)

```text
1. 业务侧在抖音开放平台创建应用 → 拿到 client_key + client_secret
2. 把应用绑定到 MCN/团长账号
3. 应用首次启动时:
   a. 把用户(MCN 账号)重定向到抖音授权页 (authorize endpoint)
   b. 用户在抖音侧确认授权
   c. 抖音回调我们一个 short-lived authorization code
   d. 用 code + client_secret 换 access_token + refresh_token
4. 后续所有 API 调用:
   a. 调 API 时把 access_token 放 header
   b. 如果 token 即将过期(剩 < 10 分钟),用 refresh_token 拿新的
   c. refresh_token 过期(30天)了,要重新走步骤 3
```

**对架构的影响**:
- 必须新增 `src/douyin/auth.ts` 文件,管 token 生命周期
- client.ts 每次调用前要检查 token 状态
- 需要持久化 token(不能进程重启就丢) → ⚠️ **mock 阶段不持久化都行,真实模式下必须接 DB 或文件**
- 需要 OAuth 回调路由 `/oauth/callback`,启动时一次性获取 token

## 四、已知 API 列表 + 我的置信度

⚠️ 抖音开放平台经历多次重构,API 路径和方法名变化频繁。下表是我整理的 OCP 用得到的 API,**真实方法名以开放平台文档为准**。

| 用途 | 推测 API 路径 / 方法名 | 我的置信度 | 备注 |
| --- | --- | --- | --- |
| **商品搜索**(精选联盟物料库) | `analytics.kolproductlist` 或 `cps/kol_product/list` | 🟡 中 | 真实路径可能是 buyin 子域下的另一组接口 |
| **商品详情** | `analytics.kolproductdetail` | 🟡 中 | 大概率存在,名字未必精确 |
| **生成推广链接(转链)** | `cps.promotion.url.create` 或类似 | 🟡 中 | 关键能力,文档应有完整描述 |
| **订单查询(订单详情/状态)** | `analytics.kolorderdataschematized` 或类似 | 🟠 较低 | 历史上多次重命名 |
| **达人粉丝/账号信息** | `oauth/userinfo` | ✅ 高 | OAuth 标准接口,可信 |

**结论**:Step 1 写 `types.ts` 时,大约 **40-50% 字段名需要拿到真实 API 后校准**。比 PDD 的不确定性高 1 个数量级。

## 五、商品模型与 OCP 协议层差异

抖音精选联盟商品有 1 个**结构性特点**前 3 家都没有:

### 商品 ≠ 单纯 SKU,是"SKU + 短视频/直播间上下文"组合

```text
传统电商:           抖音:
─────────           ────────────────────────
product_id          product_id
                    +
                    aweme_id (短视频 ID)
                    或
                    room_id (直播间 ID)
                    +
                    creator_id (推荐者达人 ID)
```

**业务表现**:
- 同一个商品在不同短视频/直播间下**佣金率不同**(品牌方按视频维度配置)
- 同一个商品被不同达人推时**短链不同**(`v.douyin.com/aaa` vs `v.douyin.com/bbb`)
- 用户点击短链进 App 时,会**唤起对应的视频或直播间**(而不仅是商品页)

### OCP 协议层需要的扩展

当前 OCP 用 3 个 descriptor pack:
- `ocp.commerce.product.core.v1` ✅ 复用即可(title / image / category / sku 等)
- `ocp.commerce.price.v1` ✅ 复用即可
- `ocp.commerce.inventory.v1` ✅ 复用即可

需要新增:
- **`ocp.commerce.livestream_ref.v1`** 🆕 描述商品挂载的短视频或直播间引用
  - 字段:`source_type: 'short_video' | 'livestream'`、`aweme_id?`、`room_id?`、`creator_id`、`creator_name`、`creator_followers?`
- **可能需要扩展 `commission` 字段语义** 🆕 抖音佣金率是"按上下文动态"的,attributes 里的 `commission_rate_bp` 需要标注"在当前 aweme/room 下"

**影响**:这不只是接一家新联盟,**是 OCP 协议从"商品目录"扩展到"内容+商品"的第一次实质性升级**。这件事的意义可能比单纯接抖音更大。

## 六、风险与不确定性清单

按风险等级排序:

### 🔴 高风险

1. **准入完全拿不到**(发生概率:中-高,如果没 MCN 资质)
   - 缓解:业务侧提前确认资质,或走 BD 绿色通道
2. **OAuth 2.0 token 刷新机制实际复杂度高于预估**(发生概率:中)
   - 缓解:Step 0.5 先做一个独立的 OAuth 流程跑通,再开 Step 1
3. **精选联盟 API 历史变更频繁,文档过时**(发生概率:中)
   - 缓解:开干前在开放平台 changelog 看最近 6 个月公告

### 🟡 中风险

4. **livestream_ref descriptor pack 推到 OCP 上游可能引发争议**(发生概率:中)
   - 不同 maintainer 对"协议是否要扩展到非传统商品"看法不同
   - 缓解:作为 PDD 后已经讨论过的"协议演化第二阶段"提出
5. **真实 fixture 与 mock 字段名差异大**(发生概率:高)
   - mock-first 模式下 Step 3 mapper 写完,真接时大概率要返工 30-40%
   - 缓解:Step 1 把 types.ts 注释里所有不确定字段名都列出来
6. **token 持久化要求引入 DB 依赖**(发生概率:高)
   - 前 3 家可以完全 stateless,抖音不行
   - 缓解:mock 阶段用内存,真实阶段接 Postgres(monorepo 已有 Drizzle)

### 🟢 低风险

7. **短链域名 `v.douyin.com` 验证不通过 OCP schema**(发生概率:低)
   - 验证:跑一遍 zod parse,基本没问题
8. **每 MCN 账号要单独 OAuth,多账号管理复杂**(发生概率:低)
   - PoC 阶段 1 个账号就够,后期再说

## 七、业务侧需要先解决的事(关键路径)

按优先级排序,业务侧不解决这些,技术侧白做:

### 🔴 必须解决(否则整个项目不可行)

1. **确定准入路径**。3 个选项里挑 1 个:
   - **A. 找 MCN 合作**:与一家小型 MCN 签合作协议,以 deeplumen 名义申请精选联盟 API。
     - ✅ 时间快(1-2 周可拿到 API)
     - ⚠️ 利益分配复杂,品牌捆绑风险
   - **B. 自己申请 MCN 资质**:
     - ⚠️ 时间慢(1-3 个月),需要旗下达人 + 营收门槛
   - **C. 团长账号路径**:
     - 🟡 比 MCN 容易,但需要现有抖音联盟流量基础
   - **D. 走字节 BD 绿色通道**:
     - 🟡 需要 deeplumen 现有业务规模背书

2. **确定 OCP 是否值得为抖音扩展 livestream_ref 协议**:
   - 这件事比单纯接抖音影响更大
   - 如果 OCP 定位是"传统电商商品目录协议",那不该扩展,应该 pivot
   - 如果 OCP 定位是"AI Agent 与所有可成交对象的协议",那应该扩展,且抖音是第一个测试场景

### 🟡 强烈建议提前做(节省时间)

3. **联系 1-2 家 MCN 摸底**,确认"如果我们 PoC 用你的 MCN 资质申请 API,能不能合作 + 商业条款"。
4. **抖音开放平台账号注册一下**,即使没有应用,先看到后台界面对申请流程有更具体感知。

## 八、技术侧并行可以做什么(不阻塞业务)

业务侧解决准入需要 2-4 周,这段时间技术可以做不依赖凭据的事:

| 任务 | 工作量 | 阻塞? | 输出 |
| --- | --- | --- | --- |
| 1. 写 OAuth 2.0 token manager(可单测) | 1 天 | 否(独立模块) | `src/douyin/auth.ts` 可复用 |
| 2. 写 `ocp.commerce.livestream_ref.v1` descriptor pack 提案 | 1 天 | 否 | 推到 OCP 上游 issue 讨论 |
| 3. 基于公开文档拟一份 types.ts(标注不确定字段) | 0.5 天 | 否 | Step 1 起点 |
| 4. 写一份对外汇报版的"抖音接入对 OCP 协议演化的意义" | 0.5 天 | 否 | 给业务/对外宣传用 |
| **合计** | **~3 天** | | 准入到位时 Step 1-2 可以**头部启动状态** |

**对比正常的 Step 1 周期(等准入 → 写代码)**:可以节省 3-5 天等待时间,但有 30% 概率写的字段名要重写。

## 九、可能的退路 / pivot 方向

如果业务侧 2 周内没能拿到准入路径,**强烈建议 pivot**,不要在准入上耗。可选方向:

| 退路 | 接入难度 | 业务价值 | 备注 |
| --- | --- | --- | --- |
| **携程联盟** | ✅ 低(个人/公司均可) | ⭐ 高(协议扩展到非电商) | 比抖音对 OCP 协议扩展的展示力更强 |
| **AliExpress 跨境** | ✅ 低(用现有 alimama 体系) | 🟡 中(海外场景) | 适合做出海 Agent |
| **快手联盟** | ⚠️ 中(类似抖音,但门槛略低) | 🟡 中(直播带货 #4) | 同样有 OAuth 复杂度,不一定省事 |
| **网易严选联盟** | ✅ 低 | 🟢 低(类目精选,体量小) | 适合作为第 4 家"备份选项" |

## 十、决策建议

### 当下立刻能做的 3 件事(我建议同时启动)

1. **业务侧**:本周内确认准入路径(A/B/C/D 选 1),给出明确时间表
2. **技术侧(我)**:接下来 3 天产出
   - `src/douyin/auth.ts` OAuth 2.0 token manager(可独立单测)
   - `docs/livestream-ref-pack-proposal.md` 协议扩展提案
   - 一份 types.ts 草稿(标注所有不确定字段)
3. **业务/技术联合**:决定 2 周后的 checkpoint
   - 如果业务侧拿到 MCN 路径 → 走 Step 1-6 完整流程
   - 如果没拿到 → 转 pivot,改接携程或 AliExpress

### 我对走不走的看法

**如果你的核心 KPI 是"接入第 4 家联盟"**:不要执着抖音,改接携程(协议扩展演化故事更强)或 AliExpress(用现有 alimama 体系,2 天搞定)。

**如果你的核心 KPI 是"OCP 协议演化到内容+商品"**:抖音必须接,准入耗 1-3 个月也值得 —— 这件事的意义是把 OCP 从"商品目录协议"升级成"Agent 与可成交对象的通用协议",战略价值远高于多接一家联盟。

**如果你的核心 KPI 是"短期对外有冲击力的 demo"**:抖音不一定最优 —— 拿到 MCN 准入慢,且单纯做"AI 推荐抖音商品"对外看起来跟前 3 家差不多。携程做"AI 帮你订机票+酒店"对外冲击力反而更大。
